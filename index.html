L1 - Physical Layer

üîå Cable Connection Problem

The Ethernet cable is damaged, improperly connected, or incompatible. The network interface card LEDs are off, no physical connection is established, and devices cannot see each other.

üîß Detailed Solution Steps

  1. Physical Check: Is the cable firmly plugged in at both ends? The RJ45 connector may not be fully inserted. Unplug and replug the cable.
  2. LED Status Analysis: Check the LEDs on the network card:
    • üü¢ Link LED (Green): If on, physical connection exists
    • üü† Activity LED (Orange): If blinking, data traffic is present
    • üî¥ If off: Cable or port is faulty
  3. Cable Test: Test with another working cable. Check if it meets Cat5e/Cat6 standards. Cables longer than 100 meters may cause signal loss.
  4. Port Change: Try a different port on the same switch/router. The port may be faulty. Test different ports on both sending and receiving devices.
  5. Cable Quality: Bending, crushing, or running the cable through sharp corners can break internal wires. Check for damaged sections along the cable.
  6. Duplex Settings: On older devices, manual full-duplex/half-duplex and speed (10/100/1000 Mbps) mismatches can occur. Enable auto-negotiation.
  7. Cross/Straight Cable: Older devices may require a crossover cable. Modern devices with Auto-MDIX handle this automatically.
  8. Final Test: Perform a ping test and verify network connectivity. Check with "ipconfig" if an IP address is assigned.
‚ö†Ô∏è Common Cable Issues:
  • RJ45 Connector Damage: Plastic clip broken, pins bent
  • Internal Wire Break: Cable excessively bent or crushed
  • Wrong Cable Type: Telephone cable (RJ11) used instead of Ethernet
  • Length Problem: Distance over 100 meters
  • Electromagnetic Interference: Interference near power cables
  • Old Cable Standard: Cat3 cable does not support gigabit Ethernet
üí° Cable Testing Tools:
  • Cable Tester: Cable continuity and pin-to-pin test
  • Network Analyzer: Signal quality and speed test
  • Multimeter: Electrical continuity check
  • TDR (Time Domain Reflectometer): Cable length and damage location

L2 - Data Link Layer

üîí Switch Port Security Violation

Port-security is enabled on the switch port and the allowed number of MAC addresses has been exceeded. The port has entered shutdown state, new devices cannot connect, and existing connections are lost. Due to a security violation, the port is in "err-disabled" state.

üîß Detailed Solution Steps

  1. Check Port Security Status: Connect to the switch and identify ports with security violations: Switch# show port-security
    Switch# show port-security interface fa0/1
    Switch# show interfaces status err-disabled
  2. Violation Details: Find out which MAC address caused the violation and the violation type: Switch# show port-security address
    Switch# show mac address-table interface fa0/1
    Switch# show logging | include PSECURE
  3. Review Security Settings: Analyze the current port-security configuration:
    • Maximum MAC Count: How many MAC addresses are allowed?
    • Violation Action: Shutdown, restrict, or protect?
    • Aging Time: How long are MAC addresses retained?
    • Sticky Learning: Are MACs learned automatically?
  4. Re-enable the Port: Manually bring up the err-disabled port: Switch(config)# interface fa0/1
    Switch(config-if)# shutdown
    Switch(config-if)# no shutdown
    Switch(config-if)# exit
  5. Clear MAC Address Table: Remove old/incorrect MAC entries: Switch# clear mac address-table dynamic interface fa0/1
    Switch# clear port-security dynamic interface fa0/1
    Switch# clear port-security sticky interface fa0/1
  6. Reconfigure Port Security: Optimize settings if necessary: Switch(config)# interface fa0/1
    Switch(config-if)# switchport port-security maximum 3
    Switch(config-if)# switchport port-security violation restrict
    Switch(config-if)# switchport port-security aging time 60
    Switch(config-if)# switchport port-security mac-address sticky
  7. Enable Automatic Recovery: Set up automatic recovery for future violations: Switch(config)# errdisable recovery cause psecure-violation
    Switch(config)# errdisable recovery interval 300
  8. Test and Monitor: Verify port status and security status: Switch# show interfaces fa0/1 status
    Switch# show port-security interface fa0/1
    Switch# ping [test-device-ip]
‚ö†Ô∏è Port Security Violation Types:
  • Shutdown: Port is completely disabled (err-disabled), manual intervention required
  • Restrict: Violating packets are dropped, port stays up, logs are kept
  • Protect: Violating packets are dropped, port stays up, no logs

L3 - Network Layer

üåê Incorrect IP Address Assignment

An incorrect IP address has been manually assigned to the device. While the network is 192.168.1.0/24, the device IP is set as 192.168.2.15. Ping tests fail and there is no network access.

üîß Solution Steps

  1. Check Current IP: Check the device's IP settings # Windows: ipconfig /all
    # Linux: ifconfig or ip addr show
  2. Network Block Detection: Learn the Router/Gateway IP and determine the correct network segment
  3. IP Correction: Select an appropriate IP from the correct IP block (192.168.1.x) and set the subnet mask (255.255.255.0)
  4. Gateway Setting: Configure the default gateway with the correct IP (usually 192.168.1.1)
  5. DNS Configuration: Add DNS server addresses (router IP or 8.8.8.8)
  6. Alternative - DHCP: Use DHCP to automatically obtain an IP instead of manual IP
  7. Test: Verify the connection with Ping ping 192.168.1.1 (gateway)
    ping 8.8.8.8 (internet)

L3 - Network Layer

üó∫Ô∏è Routing Table Error

An incorrect static route is defined on the router. Packets cannot reach the destination network and a "Destination Unreachable" error is received. There are missing or incorrect routes in the routing table.

üîß Solution Steps

  1. Check Routing Table: Review the current routing table Router# show ip route
    Router# show ip route static
  2. Identify Incorrect Route: Find routes pointing to the wrong next-hop or interface Router# show ip route 192.168.10.0
    Router# traceroute 192.168.10.5
  3. Remove Incorrect Route: Delete the incorrect static route Router(config)# no ip route 192.168.10.0 255.255.255.0 192.168.2.1
  4. Add Correct Route: Define a new route with the correct next-hop Router(config)# ip route 192.168.10.0 255.255.255.0 192.168.1.2
  5. Check Default Route: Verify the correctness of the 0.0.0.0/0 route
  6. Interface Status: Ensure next-hop interfaces are up/up Router# show ip interface brief
  7. Ping Test: Test access to the target network Router# ping 192.168.10.5
    Router# traceroute 192.168.10.5

L3 - Network Layer

Routing Protocol Issues

Problems with OSPF, EIGRP, or RIP protocols such as neighbor adjacency issues, metric calculation errors, and routing loops. Network convergence does not occur.

üîß Solution Steps

  1. Protocol Status: Check routing protocol status with "show ip protocols"
  2. Neighbor Detection: Run "show ip ospf neighbor" or "show ip eigrp neighbors"
  3. Interface Check: Review interface statuses with "show ip interface brief"
  4. Authentication: Verify authentication settings between routers
  5. Network Statements: Ensure correct network ranges are defined
  6. Metric Settings: Optimize bandwidth and delay values
  7. Loop Prevention: Check split horizon and route poisoning settings
  8. Convergence Test: Use "clear ip route *" to refresh the routing table and test convergence

show ip route
show ip ospf database
show ip eigrp topology

L5 - Session Layer

‚è±Ô∏è Session Timeout

Remote desktop connection, SSH session, or web sessions are constantly disconnecting. Session timeout values are set too low and the connection is automatically terminated when the user is idle.

üîß Solution Steps

  1. Check Current Timeout: Review session timeout settings
  2. Increase SSH Timeout: Extend timeout for console and VTY lines
  3. Enable Keep-Alive: Activate mechanisms to keep the connection alive
  4. Web Session Timeout: Adjust HTTP server session durations
  5. Firewall/NAT Timeout: Check session aging times on the firewall
  6. Client Settings: Optimize connection settings on the RDP client
  7. Test: Leave idle for a long time and test the timeout

L5 - Session Layer

Connection State Issues

Sessions are not being established or maintained properly. Potential causes include connection tracking issues, state table overflow, asymmetric routing, and firewall state inconsistencies.

üîß Solution Steps

  1. Check the connection state table (e.g., `show ip nat translations`, `show ip inspect sessions`).
  2. Verify firewall connection tracking.
  3. Adjust connection limits if necessary (e.g., `ip nat translation max-entries [num]`, `ip inspect max-incomplete high [num]`).
  4. Ensure consistent routing paths.
  5. Monitor connection establishment and teardown.
üìã Diagnostic Commands:
Router> show ip nat translations
Router> show ip inspect sessions
Router(config)# ip nat translation max-entries 10000
Router(config)# ip inspect max-incomplete high 400

L7 - Application Layer

DNS Resolution Error

Website names cannot be resolved, but access is possible via IP addresses. DNS server settings are incorrect, DNS server is unreachable, or DNS records are missing/incorrect. Users receive "Server not found" or "DNS resolution failed" errors.

üîß Detailed Solution Steps

  1. DNS Server Status: Test accessibility of primary and secondary DNS servers with ping
  2. Client DNS Settings: Check DNS settings with `ipconfig /all` (Windows) or `cat /etc/resolv.conf` (Linux)
  3. DNS Query Test: Perform manual DNS queries with `nslookup google.com` and `dig google.com`
  4. DNS Cache Cleanup: Clear cache with `ipconfig /flushdns` (Windows) or `sudo systemctl restart systemd-resolved` (Linux)
  5. Alternative DNS: Test using Google DNS (8.8.8.8, 8.8.4.4) or Cloudflare DNS (1.1.1.1, 1.0.0.1)
  6. Router DNS Forwarding: Check DNS forwarding settings and upstream DNS servers on the router
  7. Firewall Check: Verify if port 53 (UDP/TCP) traffic is blocked
  8. DNS Server Configuration: If using an internal DNS server, check zone files and records

L2 - Data Link Layer

üè∑Ô∏è VLAN Configuration Error

Devices in different VLANs cannot communicate with each other. Trunk port configuration is incorrect, VLANs are not created, or inter-VLAN routing is not set up. There is no connectivity even within the same VLAN.

üîß Solution Steps

  1. Check VLAN Status: Check existing VLANs and port assignments
  2. Create VLANs: Define missing VLANs
  3. Access Port Configuration: Assign ports to the correct VLAN
  4. Trunk Port Setup: Configure trunk between switches
  5. Inter-VLAN Routing: Set up routing with router-on-a-stick or Layer 3 switch
  6. Native VLAN Check: Verify native VLAN compatibility on trunk ports
  7. Test: Perform ping tests within and between VLANs
L3 - Network Layer

üì° DHCP Server Error

New devices cannot connect to the network and do not receive an IP address. The DHCP server IP pool is exhausted, the service is stopped, or DHCP relay settings are incorrect. Clients work with manual IP but cannot get an automatic IP.

üîß Solution Steps

  1. DHCP Server Status: Check if the service is running
  2. IP Pool Capacity: Check available IP addresses
  3. Enable DHCP Service: Make sure the service is active
  4. Reconfigure Pool: Expand or redefine the IP pool
  5. Excluded Addresses: Exclude IPs that need to be reserved
  6. DHCP Relay: Set helper-address for clients in different subnets
  7. Client Test: Renew DHCP on the client
